## 1. 랜딩페이지 구조 (Channel.io)

### 필수 구조: Hero → Details → CTA → References

| 섹션 | 역할 | 핵심 원칙 |
|---|---|---|
| **Hero** | 첫 인상, "여기 맞아" 확인 | 광고/유입 경로와 메시지 일치시킬 것 |
| **Details** | 제품/서비스 상세 설명 | Hero의 주장을 뒷받침 |
| **CTA** | 전환 유도 | **최대 2개**, 같은 퍼널로 유도 |
| **References** | 약관, 신뢰 요소 | 유저의 망설임 줄이기 |

### CTA 버튼 원칙
- "지금 구매" ❌ → "무료로 시작하기" ✅ (부담 낮추기)
- "가입하기" ❌ → "준비하기" ✅ (행동 가볍게 표현)
- 차가운 유저에게는 "더 알아보기"가 "구매하기"보다 효과적
- 버튼 2개까지 OK, 단 같은 퍼널 목적지로

### 치명적 실수들
- **디바이스 불일치**: 모바일 광고 → 데스크톱 랜딩 (반응형 필수)
- **메시지 불일치**: 유입 메시지와 랜딩 내용이 다름
- **액션 부재**: 버튼이나 다음 단계가 없음
- **정보만 나열**: 전환 기회 자체가 없음

### 테스트 & 최적화
- 한 번에 하나씩 수정 (위→아래 순서)
- 테스트당 3~14일 데이터 수집
- 순서: Hero 강화 → Detail 보강 → CTA 개선

## 2. 마케팅 문구 클릭률 올리는 6가지 원칙 (Toss)

### 원칙 1: 하나의 핵심 메시지만
- 여러 혜택 나열 ❌ → 하나의 명확한 행동 ✅
- 예: "소비 궁합 알아보고 신혼부부 지원금 받기" → "커플 유형 테스트, 문제 10개 찍고 결과 보기"
- **결과: 클릭률 10배 상승**

### 원칙 2: 작지만 확실한 보상
- "최대 100만 원" ❌ → "최소 100원 확정" ✅
- 불확실한 큰 보상보다 확실한 작은 보상이 효과적
- **결과: 노출 대비 20배 차이**

### 원칙 3: 행동 표현을 가볍게
- "가입하기" → "준비하기"로 바꾸면 심리적 부담 감소
- 유저가 느끼는 노력의 크기를 줄여주는 단어 선택

### 원칙 4: 정보 유형 명시
- "새로 나온" 추가 → 클릭 6배 증가
- "목록", "모아보기" → 가치를 명확하게 전달

### 원칙 5: 조건과 노력 구체화
- "미션" ❌ → "금융 미션 4개" ✅ (전환율 4배)
- "빈칸 채우기" ❌ → "8칸 채우고" ✅ (클릭 2배)
- 구체적 숫자가 망설임을 줄임

### 원칙 6: 익숙한 일상 행동 환기
- "정답 보기" ❌ → "정답 찍기" ✅
- 물리적/일상적 동작을 연상시키는 표현이 클릭+전환 모두 향상

## 3. 실험 설계 가이드 (Toss — 신입 디자이너용)

### 우선순위 매트릭스
문제가 여러 개일 때, 각각을 평가:
- **반복 속도**: 법적/컴플라이언스 검토 없이 빠르게 테스트 가능한가?
- **비즈니스 임팩트**: 핵심 지표에 의미 있는 영향을 주는가?

### 실험 설계 체크리스트
1. 퍼널을 체계적으로 분해하고, 속도 × 임팩트로 우선순위 결정
2. 가설 수립 전에 맥락을 깊이 파악
3. 과거 실험의 "문제 정의"를 참고 (결과만 보지 말 것)
4. 솔루션 설계 전에 성공 지표 정의
5. 가설이 디자인에 명확히 드러나는지 확인
6. 실험당 의미 있는 변수 하나만 테스트

### 핵심 교훈
- 실험 2: 기능 중심 → 가치 중심 카피로 변경 → **CVR 3%+ 향상**
- 실험 3: 구체적 시나리오를 상상하게 하는 표현 → **CTR 5%+ 향상**
- 빠른 반복보다 **명확한 가설 수립**이 더 중요

## 4. 유저 리서치 — UX 마찰 제거 (Toss)

### 사례: 외국인 인증 실패 원인 분석

#### 발견된 핵심 UX 문제들
1. **이름 입력 복잡성**: 포맷 불일치로 8년간 인증 실패한 유저도 있음
2. **주소 입력 장벽**: 한글 타이핑 어려움 + 검색 결과 과다
3. **누적 마찰**: 온보딩의 작은 문제들이 쌓여 앱 자체를 포기하게 만듦

#### 적용 원칙
- 실패 원인을 명확히 알려줄 것 (왜 안 되는지 설명)
- 입력 포맷을 유연하게 수용할 것
- 온보딩 단계마다 마찰을 최소화할 것
- **결과: 외국인 인증 성공률 약 15% 향상, 내국인과 격차 해소**

### Onchain Korea 적용 포인트
- 블록체인 초보자 = 외국인 유저와 유사한 상황 (새로운 용어, 복잡한 프로세스)
- 지갑 생성, 시드 구문 백업 등의 단계에서 **"왜 실패했는지"** 명확히 안내
- 각 단계의 마찰을 최소화하는 Action Guide 설계

## 5. 브랜드 페르소나 디자인 (Toss)

### 핵심 기준 4가지
1. **단순함 유지** + 매력과 완성도 향상
2. **명확한 브랜드 표현**: 얼굴 비율로 "똑똑하고 빠르고 신뢰할 수 있는" 인상 전달
3. **글로벌 포용성**: 단일 피부톤 ❌ → 이모지 표준 기반 5가지 피부톤
4. **확장 가능한 일관성**: 아이콘~광고까지 다양한 크기에서 일관된 인상

### 적용 포인트
- 캐릭터/일러스트 사용 시, 친근함과 전문성의 균형
- 하나의 톤이 아닌 다양한 표현 수용
- 작은 아이콘에서도 명확하게 보이는 디자인

## 6. 컬러 시스템 설계 (Toss — 7년만의 업데이트)

### 문제점
- 같은 토큰값인데 색상마다 밝기가 달랐음 (Grey 100 ≠ Red 100 체감 밝기)
- 다크모드에서 라이트모드와 명도 기준 불일치
- 저해상도 디스플레이에서 WCAG 접근성 미달

### 해결: OKLCH 컬러 스페이스
- 인간의 색상 인지에 기반한 균일한 밝기 스케일링
- 자동 대비 준수 + 커스텀 브랜드 색상 자동 생성
- 단, 노란색은 수학적 계산만으로 부족 → 수동 보정 필요

### 토큰 구조: 3-레이어 네이밍
```
Target (fill, text, border) + Role (brand, neutral, primary) + Variant (weak, alt)
```
- 디자인 결정 부담 감소
- 일관된 의도 매핑

### 테마 시스템
- 비즈니스 버티컬별 별도 디자인 시스템 ❌
- 시맨틱 토큰 오버라이드로 파생 테마 생성 ✅
- 예: 트레이딩 앱에서는 빨간색이 "긍정" 컬러 (주식 상승)

### Onchain Korea 적용 포인트
- 현재 navy/accent/hot 3색 체계 → 시맨틱 토큰으로 확장 고려
- 다크모드 지원 시 OKLCH 기반 색상 설계
- 접근성(WCAG) 대비 검증
